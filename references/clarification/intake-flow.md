# 澄清流程规范

[返回总入口](../../SKILL.md) · [模糊输入识别](./ambiguity-gate.md) · [问题重述](./problem-restatement.md) · [防漂移规则](./focus-anchor.md)

## 一、文件作用

这个文件只解决一件事：

- 当 `ambiguity-gate.md` 判断“还不能正式分析”后，系统应该如何分轮澄清。

它不负责：

- 判断是否需要澄清
- 设计问题重述模板
- 输出正式分析

对应关系：

- 是否进入澄清：看 [ambiguity-gate.md](./ambiguity-gate.md)
- 澄清后的问题重述格式：看 [problem-restatement.md](./problem-restatement.md)
- 长问题如何防迷失：看 [focus-anchor.md](./focus-anchor.md)

## 二、调用条件

只有满足下面任一情况，才进入澄清流程：

- 用户主要在表达情绪，没有给出具体事件
- 用户只给结论，没有过程
- 关键人物或关系结构不清楚
- 用户目标不清楚，不知道是想推进、修复、止损、退出还是判断
- 时间线混乱，无法判断当前阶段
- 如果直接分析，会明显建立在大量猜测上
- 问题属于复杂多方问题，但关键结构还没展开

如果问题已经足够清晰，不进入澄清，直接分析。

## 三、先区分普通问题和复杂问题

### 1. 普通问题

通常表现为：

- 单点卡住
- 人物不多
- 关键事件明确
- 主要是在判断下一步怎么做

这类问题常见 `1` 到 `2` 轮澄清就够。

### 2. 复杂问题

通常表现为：

- 涉及三方及以上关键人物或组织
- 表面像双人问题，但高度可能还牵涉第三方、家庭系统、旧关系、利益因素、健康因素、隐性事件或长期积累的问题
- 同时牵涉关系、执行、利益、权利、资源、对外主体等多个层面
- 用户已经怀疑“另起一套”“结构分叉”“共同体在被抽空”
- 出现新公司、新账号、新项目壳、新收款主体、新群组、新对外名义等并行容器
- 一条事件背后可能还连着更长的阶段变化

这类问题不要按“默认三问、最多五问”去机械处理，而要按轮次逐步问开，直到结构基本看清。

## 四、澄清目标

澄清不是为了把所有细节问全，而是为了把问题问到“足够支撑正式分析”。

进入正式分析前，优先补齐这些信息：

1. 目标：用户最想解决什么
2. 事件：最近一次关键事件，以及必要时的关键事件链
3. 人物：这件事里谁最关键，谁和谁有关联
4. 权利与资源：关键的信息、决策、客户、合同、现金流、资源抓手分别在谁手里
5. 尝试：已经做过什么，结果怎样
6. 约束：最担心什么，最不能接受什么
7. 缺口：还有哪些未确认点会直接影响正式判断

如果问题偏长、偏复杂、用户一次问了多个问题，澄清目标还要多两项：

8. 先冻结主问题和本轮回答范围
9. 先建立 `原始问题对照 + 案件工作单`

## 五、澄清节奏

### 1. 单轮负载

- 每轮只问 `1` 到 `2` 个高价值问题。
- 复杂问题在必要时可问 `3` 个，但不要更多。
- 不要把澄清写成审讯式清单。

### 2. 轮次原则

- 普通问题常见 `1` 到 `2` 轮就够。
- 复杂问题允许多轮澄清，不设僵硬的总题数上限。
- 每轮答完后，都要重新过一次 `ambiguity-gate.md`。
- 只要关键结构还没展开，就继续澄清，不要抢先进入正式分析。
- 如果问题偏长、偏复杂、用户一次问了多个问题，或已经进入第二轮澄清，就同步启用 `focus-anchor.md`。
- 一旦启用 `focus-anchor.md`，不要先继续漫谈；先补 `主问题锚点 -> 原始问题对照 -> 案件工作单`，再决定提哪 `1` 到 `2` 个问题。

### 3. 复杂问题优先补的结构位

复杂问题优先补下面这些位置：

- 关键人物/组织图谱
- 隐性第三方或隐性原因是否存在，例如父母、闺蜜、前任、财务、性与身体、外部关系、历史旧账
- 关键事件链，而不是单点情绪事件
- 信息权、决策权、客户、现金流、对外主体、资源抓手现在大致在谁手里
- 是否出现新容器、新公司、新系统、新账号、新项目壳
- 用户真正要守住、修复、推进或止损的东西是什么

## 六、提问顺序

如果是长问题、复杂问题或多问并行问题，提问前先做一个固定动作：

1. 先冻结主问题锚点
2. 再列出原始问题对照
3. 再检查案件工作单缺哪一栏
4. 再从缺口里挑本轮最高价值的 `1` 到 `2` 个问题

不要直接从用户原始长文里顺着戏剧性最强的细节追问。

默认按下面顺序补信息：

1. 先问目标
2. 再问最近一次关键事件
3. 再问关键人物与关系
4. 如果问题复杂，再问关键权利、信息、资源和容器分布
5. 再问已做尝试
6. 最后问约束、底线和仍未确认的关键缺口

如果某项已经很清楚，就跳过，不机械重复。

## 七、推荐问法

### 1. 目标

- 你现在最想推进的结果是什么？

### 2. 关键事件

- 最近一次最能说明问题的具体事件是什么？

### 3. 关键人物

- 这件事里最关键的人分别是谁？他们各自最在意什么？

### 4. 权利与资源分布

- 现在最关键的信息、决策、客户、合同、现金流或对外口径，分别掌握在谁手里？

### 5. 新容器或并行系统

- 最近有没有出现新的公司、账号、项目壳、收款主体、对外名义或群组，让原来的共同体开始分叉？

### 6. 已做尝试

- 你已经试过什么方法？结果怎么样？

### 7. 约束与底线

- 你现在最担心什么，或者最不能接受的结果是什么？

## 八、每轮结束后的动作

每轮澄清后，只做下面六件事：

1. 简要整理这轮新增了哪些关键信息
2. 重新判断还有哪些结构位没展开
3. 决定是继续澄清，还是进入问题重述
4. 如果用户已经提到 `HTML`，或当前问题天然容易让用户期待最终会落成 `HTML 报告`，要顺手做一句预期管理，明确说明当前信息还不够生成 `HTML`
5. 如果当前问题已经偏长或用户一次问了多个问题，更新一次 `主问题锚点 / 原始问题对照 / 案件工作单 / 轮次日志`
6. 重新确认下一轮是否仍只需要回答 `1` 到 `2` 个问题

如果仍然存在下面任一情况，就继续澄清：

- 关键人物没齐
- 隐性第三方或隐性原因还没排清
- 关键事件链没对齐
- 关键权利或资源分布不清
- 疑似有新容器，但关系未展开
- 仍无法判断用户到底要推进、修复、止损还是退出
- 还不能稳定填出 `案件工作单`
- 用户原始问题很多，但这轮先回答哪几题还没冻结

## 九、什么时候不能进入正式分析

即使已经问过一两轮，只要还存在下面任一情况，就不要进入正式分析，更不要进入 HTML：

- 局面仍主要靠猜
- 复杂问题还没问清关键结构
- 还说不清谁和谁在组成当前局面
- 还说不清是否存在足以改写结论的第三方、旧关系、家庭或身体因素
- 还说不清真正要守住的是什么
- 还说不清最关键的代价和底线是什么
- `主问题锚点 / 原始问题对照 / 案件工作单` 还没成形

这时正确动作是继续问，或者先明确告诉用户：当前只能先收束已知结构，还不能正式展开分析。

如果用户已经提到 `HTML`，这里还要把话说满，不要只说“先补一点信息”，而要明确说：

- 当前仍在澄清阶段
- 现有信息不足以支撑正式成品
- 因此现在不能生成 `HTML 报告`
- 等关键缺口补齐后，再进入正式输出

如果问题已经偏长，还要同时说明：

- 这轮先解决哪 `1` 到 `2` 个问题
- 哪些问题仍待后续回答

## 十、收束动作

当已经能稳定重述问题时，先做一次简短归纳。

推荐格式：

```text
主问题锚点：
- 现在最要解决的是……
- 这轮先回答……
- 暂不展开……

原始问题对照：
- Q1：……
- Q2：……

案件工作单：
- 目标：
- 关键事件链：
- 关键人物/对象：
- 关键权利或资源分布：
- 已做尝试：
- 约束/底线：
- 已确认事实：
- 当前推断：
- 未证实传闻：
- 仍待确认：

我目前先确认到这几件事：
- 你的目标是……
- 目前已经确认的关键事件链是……
- 关键人物/对象包括……
- 现在最关键的权利/资源/关系位置在……
- 你已经试过……
- 当前主要约束是……
- 仍待确认的是……

如果这版没有偏，我再进入正式分析。
```

如果还是复杂问题，并且“仍待确认”的部分会改写结论，就不要进入正式分析，继续澄清。

如果用户已经提过 `HTML`，可以在这段收束后补一句类似的话：

```text
现在还差 [关键缺口] 没补齐，所以我先不生成 HTML 报告；等这些点明确后，再把它整理成正式成品。
```

## 十一、进入分析前的最小整理格式

澄清结束后，至少整理成下面三块：

```text
主问题锚点：
原始问题对照：
案件工作单：
```

其中 `案件工作单` 至少整理成下面十栏：

```text
目标：
关键事件链：
关键人物/对象：
关键权利或资源分布：
已做尝试：
约束/底线：
已确认事实：
当前推断：
未证实传闻：
仍未确认：
```

只有 `主问题锚点` 已冻结、`原始问题对照` 已可跟踪、`案件工作单` 前九栏已经基本成立，并且“仍未确认”不会推翻主判断时，才进入正式分析。

## 十二、禁止事项

- 不要把澄清理解成“默认三问、最多五问”的僵硬问卷。
- 不要在复杂多方问题里，只问一轮就往下输出正式分析。
- 不要一边追问，一边偷跑大段结论。
- 不要把用户的表面情绪直接当成核心问题。
- 不要追问与判断无关的枝节细节。
- 如果澄清还没闭合，不要提前问用户要文字版还是 HTML 版。
- 如果信息还不够，不要提前生成 `HTML 报告`，也不要用模糊说法让用户误以为报告马上就会出来。
- 不要在长问题里只补信息、不固定主问题；这样后面很容易越聊越偏。
- 不要在没有 `主问题锚点 / 原始问题对照 / 案件工作单` 的情况下，直接写长篇澄清总结。
